The Long Road Here

Computation didn't begin with electricity. It began with the need to count.

The word computer originally referred to a human being — a person whose job was to perform calculations. Armies of human computers computed artillery tables. Banks employed them to calculate interest. The machine we now call a computer was built to replace that person.

~3000 BC
The Abacus
The first mechanical aid to calculation. Physical beads represent numbers. Humans still do the logic; the tool just stores intermediate state.
1642
Pascal's Pascaline
Blaise Pascal built the first mechanical calculator using gears. Turning a wheel carried values from one digit to the next — mechanical addition.
1837
Babbage's Analytical Engine
Charles Babbage designed — but never fully built — a steam-powered machine with an input (punch cards), a mill (the CPU), a store (memory), and output. The entire architecture of modern computers, 100 years early.
1843
Ada Lovelace — First Algorithm
Ada Lovelace wrote the first published algorithm intended for a machine. She understood that the Analytical Engine could manipulate symbols — not just numbers — making her arguably the first to conceive of software.
1936
Alan Turing — The Universal Machine
Turing published his paper on "computable numbers," describing a theoretical machine that could simulate any other machine by reading and writing symbols on a tape. This "Turing machine" is the conceptual blueprint for every computer ever built.
1945
ENIAC — First Electronic Computer
30 tons. 18,000 vacuum tubes. 150kW of power. ENIAC could do 5,000 additions per second. Programmed by physically rewiring cables — no software, just hardware configuration.
1947
The Transistor
Bell Labs invented the transistor — replacing hot, fragile, power-hungry vacuum tubes with a tiny solid-state switch. This is the moment everything changed.
1958
Integrated Circuit
Jack Kilby at Texas Instruments placed multiple transistors on a single piece of silicon. A chip was born. The path to billions of transistors on a fingernail was opened.
1971
Intel 4004 — First Microprocessor
An entire CPU on a single chip. 2,300 transistors. The computer shrinks from a room to a rectangle smaller than your thumbnail.
Today
Apple M-series / AMD Ryzen / etc.
Over 100 billion transistors on a chip the size of a postage stamp. Each transistor is ~3 nanometers — roughly 20 silicon atoms wide.

What Is a Computer?

Strip away every abstraction. What remains?

A computer is a machine that reads instructions stored in memory, executes those instructions to transform data, and writes results back to memory. That's the complete definition. Everything else — your browser, your photos, your music — is an elaborate stack of abstractions built on top of that single loop.

The Von Neumann Architecture (1945)

John von Neumann described the architecture used by virtually every computer today: a CPU, memory, and I/O devices connected by a bus. The key insight was that programs and data live in the same memory — code is just data that happens to be instructions. This "stored-program" concept is what lets you run any program without rewiring anything.

Every action your computer takes — playing a video, running a game, displaying this page — reduces to millions of tiny steps through this loop: Fetch an instruction → Decode it → Execute it → Store the result → Repeat.

Binary: The Language of Voltage

Why do computers speak in 1s and 0s? Because voltage is either high or low.

Deep inside a chip, logic is implemented as electrical voltage levels. A transistor is either conducting (~1.8V, call it 1) or not conducting (~0V, call it 0). There is no third option in digital logic. This maps perfectly to binary — base-2 arithmetic.

Everything a computer processes — text, images, music, video, programs — is ultimately encoded as sequences of bits (binary digits). A bit is a single 0 or 1. Eight bits make a byte. Your 1TB hard drive holds ~8 trillion individual bits.

When you type the letter "A", the keyboard sends the binary number 01000001 (decimal 65) to the computer. The computer stores it in memory as that pattern of 8 voltage levels — high-low-low-low-low-low-high — and later your screen interprets those same bits to know to light up the pixels that form the shape of "A".

Why not use base-10?

Theoretically you could build a decimal computer — some early machines were decimal. But reliably distinguishing 10 distinct voltage levels (0.0V, 0.2V, 0.4V... 1.8V) in a real circuit subject to noise, temperature drift, and manufacturing variation is extraordinarily difficult. Two levels (high/low) is robust and simple. Binary wins by physics.

The Transistor

The most important invention in human history. A switch too small to see.

A transistor is a three-terminal semiconductor device. Think of it as an electrically controlled switch. Apply voltage to the Gate terminal, and current flows between Source and Drain. Remove the gate voltage, and current stops. On and off. 1 and 0.

  MOSFET (Metal-Oxide-Semiconductor Field-Effect Transistor)
  — the type used in modern CPUs —

  Gate (control voltage)
  │
  ────┼────
  Source  │ │  Drain
  ───────[   ]───────
  Substrate
  │
  GND

  Gate voltage LOW  →  No current flows  →  OFF (0)
  Gate voltage HIGH →  Current flows     →  ON  (1)

Modern CPUs use a variant called FinFET (a 3D transistor that wraps the channel on three sides), now evolving to Gate-All-Around (GAA) where the gate wraps entirely around the channel. This allows transistors to shrink to 3nm and below.

Scale Check

A human hair is ~70,000 nanometers wide. A modern transistor is 3nm. You could fit roughly 23,000 transistors across a single human hair. The M3 Ultra chip contains about 153 billion transistors on a chip you can hold between two fingers.

Moore's Law

In 1965, Gordon Moore observed that the number of transistors on a chip doubled approximately every two years. This "law" held remarkably well for 50 years, driving an exponential increase in computing power at decreasing cost. It is now slowing — we are approaching physical limits (a few atoms wide) — which is why chip designers are turning to 3D stacking, chiplets, and specialized accelerators (GPUs, NPUs) instead of just shrinking transistors further.

The Half Adder — Where Math Emerges from Logic

Connect an XOR gate and an AND gate to the same two inputs. The XOR gives you the sum bit, the AND gives you the carry bit. You have just built a circuit that can add two single-bit numbers. Chain 64 of these together and you can add two 64-bit integers — the basic arithmetic your CPU does billions of times per second. All math emerges from gates. All gates emerge from transistors. All transistors emerge from voltage levels. Voltage levels are physics.

The NAND gate is special: it is functionally complete, meaning any possible logic function can be built from NAND gates alone. AND, OR, NOT, XOR, adders, multiplexers, flip-flops — all from one type of gate made from two transistors.

The CPU — Mind of the Machine

A CPU is a loop that runs three to five billion times per second.

The Central Processing Unit is an enormously complex collection of logic gates organized to execute instructions. At its heart is the Fetch → Decode → Execute cycle, repeated billions of times each second at the pace dictated by the clock signal — a crystal oscillator that ticks like a metronome.

The Fetch–Decode–Execute Cycle

📥 Fetch
Read next instruction from memory using the Program Counter register
🔍 Decode
Interpret the binary instruction — what operation? which registers?
⚙️ Execute
ALU performs the operation (add, compare, shift, etc.)
💾 Memory
Read from or write to RAM if needed
✍️ Write Back
Store result into a register, advance Program Counter

Key Components Inside a CPU

ALU (Arithmetic Logic Unit): Performs addition, subtraction, bitwise AND/OR/NOT/XOR, shifts, and comparisons. It is an array of adders and logic gates. No magic — just gates.

Registers: Tiny, ultra-fast storage directly on the CPU die. A modern 64-bit CPU has ~32 general-purpose registers (e.g., RAX, RBX in x86). Operations happen in registers. Reading from RAM is far too slow — you must load values into registers first.

Control Unit: The orchestrator. Reads the instruction, activates the right parts of the ALU, manages register reads/writes, controls the data bus. It's essentially a state machine.

Program Counter (PC): A register that holds the memory address of the next instruction to execute. The CPU blindly follows this pointer through memory, executing whatever it finds there. A program is just a sequence of instructions laid out in memory, and the PC marches through them in order (unless a jump or branch instruction redirects it).

Clock: A quartz crystal vibrates at a precise frequency — say, 4 GHz = 4 billion vibrations per second. Every cycle, logic gates settle to their new values. The clock keeps everything synchronized. Higher clock = faster, but also more heat (power = capacitance × voltage² × frequency).

Modern CPU Tricks — Why Your CPU is Scary Fast

Pipelining

Instead of completing one instruction before starting the next, a pipelined CPU works on multiple instructions simultaneously — like an assembly line. While stage 4 (memory access) processes instruction N, stage 3 (execute) is already working on instruction N+1, stage 2 (decode) on N+2, and so on. Modern CPUs have pipelines 14–20 stages deep.

Out-of-Order Execution

The CPU doesn't necessarily execute instructions in the order you wrote them. If instruction 5 is waiting for memory (slow) but instruction 6 can already execute (its inputs are ready), the CPU executes 6 first. It reorders execution for maximum throughput, then presents results in the correct order. This is happening right now inside your CPU, thousands of times per second.

Branch Prediction

Your code is full of if-statements. When the CPU hits a branch, it doesn't know yet which path to take — it hasn't computed the condition yet. Rather than stall, it guesses based on history. If its prediction was wrong, it discards the speculative work. Modern branch predictors are right ~95–98% of the time. (This is also the basis of the Spectre CPU vulnerability — speculative execution that leaks secrets.)

Superscalar Execution

Modern CPUs have multiple ALUs and can execute several instructions literally simultaneously within a single clock cycle. A modern core might retire 4–6 instructions per clock. Multiply by 3GHz and 8 cores and you're at hundreds of billions of operations per second.

Memory — Where Everything Lives

Memory is a hierarchy. Speed costs. The closer to the CPU, the faster — and smaller.

The CPU is extraordinarily fast. RAM is comparatively very slow (~100ns latency vs <1ns for registers). This speed mismatch is the central challenge of computer architecture. The solution: a memory hierarchy where fast/small/expensive caches sit between the CPU and slow/large/cheap main memory.

Registers
~32 × 8 bytes  ·  <0.3ns  ·  on CPU die
L1 Cache
32–128 KB per core  ·  ~1–4 cycles  ·  on CPU die
L2 Cache
256 KB–4 MB per core  ·  ~10–20 cycles  ·  on CPU die
L3 Cache (shared)
8–64 MB  ·  ~40–60 cycles  ·  on CPU die
RAM (Main Memory)
8 GB – 192 GB  ·  ~100ns / 300–500 cycles  ·  separate chip

When the CPU needs data, it first checks L1. If not there (cache miss), it checks L2, then L3, then RAM. A cache miss to RAM takes ~300 CPU cycles — in that time, the CPU could have executed hundreds of instructions. The entire field of "cache-friendly programming" is about arranging data so the CPU finds it in cache as often as possible.

How RAM Physically Works

Dynamic RAM (DRAM) stores each bit as charge in a tiny capacitor. Charged = 1, discharged = 0. Because capacitors leak, DRAM must be refreshed every ~64 milliseconds — the memory controller continuously re-reads and re-writes every bit in RAM just to maintain it. Your RAM is continuously rewriting itself thousands of times while you read this.

The Stack and the Heap

RAM is logically divided. The stack is a fast, automatically managed region used for function calls — local variables, return addresses. It grows and shrinks as functions call each other. The heap is a larger region used for dynamic allocation (malloc / new) — anything that needs to persist beyond a single function or whose size isn't known at compile time. Memory leaks happen when heap allocations are never freed.

08 / 12

Storage — Permanent Memory

RAM forgets everything when the power goes off. Storage doesn't.

There are two kinds of memory: volatile (loses data without power — RAM) and non-volatile (persists — storage). When you save a file, data moves from RAM to storage. When you open a file, it moves back.

Hard Disk Drive (HDD): Magnetic platters spin at 5,400–7,200 RPM. A read/write head moves across the platter surface, magnetizing tiny domains (north = 1, south = 0). Accessing data requires physically moving the head — mechanical latency of ~8ms. Fast for sequential reads; slow for random access.

Solid State Drive (SSD): No moving parts. Data is stored in NAND flash cells — transistors with a floating gate that trap electrons. Trapped electrons = 1, empty = 0. No mechanical seeking. Random read latency drops to ~50–100μs — 100× faster than HDD. Modern NVMe SSDs can read at 7 GB/s.

File Systems: Raw storage is just a sequence of blocks. The file system (NTFS, ext4, APFS, etc.) is a data structure — a tree of directories and files — that maps human-readable paths to physical block locations. When you delete a file, the OS often just removes the directory entry and marks those blocks as reusable; the actual data may persist until overwritten (the basis of data recovery).

The Operating System

The invisible manager. You never see it, but it controls everything.

Without an OS, every program would need to know exactly how to talk to every piece of hardware. The OS solves this by being the sole mediator between programs and hardware. It provides abstractions: instead of writing to specific memory addresses on a graphics card, your program calls draw() and the OS handles the rest.

Applications  (Chrome, Photoshop, your code) User Space
System Libraries  (libc, Win32 API, Cocoa) Abstraction
System Calls  (read, write, open, fork, mmap) Kernel Interface
OS Kernel  (process mgmt, memory mgmt, scheduling, drivers) Kernel Space
Hardware  (CPU, RAM, Disk, Network, GPU) Physical Layer

What the Kernel Actually Does

Process Management: Your CPU has N cores. You have hundreds of running programs. The OS scheduler rapidly switches each core between programs — so fast that every program feels like it's running continuously. This is time-sharing. The scheduler decides which process runs next, for how long, and on which core.

Virtual Memory: Every process believes it has the entire address space to itself — say, a 48-bit space of 256 terabytes. In reality, only the pages currently in use are backed by actual RAM. The Memory Management Unit (MMU) — hardware in the CPU — translates virtual addresses to physical RAM addresses using page tables. This gives every process isolation and lets the OS swap cold pages to disk.

System Calls: User programs cannot directly access hardware. They must ask the OS via system calls. read(), write(), open(), fork(), exec(), mmap() — these are the ~300 doorways between user programs and the kernel. A system call switches the CPU from user mode to kernel mode (a hardware privilege level), the kernel performs the action, and switches back.

Interrupts: When you press a key, the keyboard sends an electrical interrupt signal to the CPU. The CPU stops whatever it's doing, saves its state, and jumps to the OS's interrupt handler, which reads the keypress and returns. This is how all I/O works — the CPU doesn't poll devices; devices signal the CPU when they have data.

From Code to Execution

How do your words become electricity?

When you write x = 5 + 3 in Python, that text goes through an extraordinary journey before transistors actually switch.

  Your Python code:       x = 5 + 3
  │
  ▼
  Python Interpreter      Reads text, parses it into an Abstract Syntax Tree
  (or Compiler)           "LOAD_CONST 5 → LOAD_CONST 3 → BINARY_ADD → STORE x"
  │
  ▼
  Bytecode / IR           Platform-independent intermediate representation
  │
  ▼
  Machine Code            x86-64 / ARM instructions (ADD RAX, 5 / ADD RAX, 3)
  │               Binary opcodes: 01001000 10000011 11000000 00000011
  ▼
  CPU Microcode           Translates complex instructions to micro-ops
  │               (Modern CPUs translate x86 instructions into
  ▼                simpler internal RISC-like micro-operations)
  Logic Gates             AND, OR, XOR, adders perform the actual addition
  │
  ▼
  Transistors             Voltage levels switch, current flows
  │
  ▼
  Electrons               Physics. Electromagnetism. Reality.

Compiled languages (C, Rust, Go) skip the interpreter and translate directly to machine code at build time — faster because there's no runtime translation overhead. Interpreted languages (Python, JavaScript) translate during execution. JIT (Just-In-Time) compilers (used in modern JavaScript engines and the JVM) detect hot paths and compile them to native code at runtime — a hybrid approach.

The Instruction Set Architecture (ISA)

The ISA is the contract between hardware and software — the set of instructions the CPU understands. x86-64 is used in most laptops and desktops. ARM is used in smartphones, Apple Silicon Macs, and most embedded devices. A program compiled for x86 cannot run natively on ARM (hence Rosetta 2, Apple's translation layer). The ISA is the language the CPU speaks.

Networking — Computers Talking

The internet is a system for moving packets of bits between machines.

All networking reduces to the same thing: packing data into packets (chunks of bits with a header indicating source, destination, and how to reassemble them), routing those packets through a network of routers, and reassembling them at the destination.

What happens when you type a URL and press Enter

1. Browser parses URL → needs IP address for "google.com"
↓
2. DNS lookup → sends UDP packet to DNS server → returns 142.250.x.x
↓
3. TCP handshake → SYN → SYN-ACK → ACK (3 packets to establish connection)
↓
4. TLS handshake → exchange cryptographic keys, verify certificate (HTTPS)
↓
5. HTTP GET request → "GET / HTTP/2" sent as TCP packets
↓
6. Packets travel through routers (each router reads destination IP, consults routing table, forwards)
↓
7. Server processes request, sends back HTML in TCP packets
↓
8. Your OS TCP stack reassembles packets in order, acknowledges receipt
↓
9. Browser parses HTML → requests CSS/JS/images (more TCP connections)
↓
10. Browser renders (builds DOM tree → layout → paint → GPU composites pixels on screen)

All of this — typically in under 200 milliseconds. The entire internet is built on layered protocols: physical (electricity/light), data link (Ethernet/WiFi), network (IP), transport (TCP/UDP), and application (HTTP/DNS/SMTP). Each layer knows only about the layer above and below it.

The Most Important Insight

A computer has no understanding of what it is doing. It executes instructions. Billions of them. Very fast. The intelligence — the meaning — is injected entirely by the programmer who arranged those instructions, and by the humans who interpret the output. A computer is a very fast, very obedient, completely literal instruction-follower with no judgment whatsoever. Everything that seems intelligent about computers is a structure humans built on top of this blank substrate.

The obscure is now obvious. You are not looking at magic. You are looking at the most elaborate engineering project in human history — built from almost nothing, one layer of abstraction at a time, by thousands of people across decades. And it all starts with a tiny switch made of sand.

Built from sand → silicon → transistors → gates → CPU → OS → software → you
Every layer reveals another layer. There is no bottom.
